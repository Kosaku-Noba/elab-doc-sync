## 1. FR-15 タグ・メタデータ管理

### 1.1 タグ同期方針: 追記のみ

push 時の YAML `tags` 設定は **追記のみ** で動作する。リモートに存在するが YAML に記載のないタグは外さない。

**理由:**
- eLabFTW 上で手動付与されたタグや他システム由来のタグを保護する
- 「完全置換」は破壊的であり、事故コストが高い
- タグを外したい場合は `esync tag remove` で明示的に操作する

**代替案（却下）:**
- 完全置換モード — Codex レビューで P1 指摘を受け、追記のみに変更

### 1.2 メタデータ保全方針

`get_metadata()` は eLabFTW の metadata フィールドが null・不正 JSON・非 dict の場合に空 dict を返す。

**理由:**
- eLabFTW の metadata フィールドは通常 JSON object 文字列または null
- 不正な状態のデータを無理にパースするより、安全なデフォルトを返す方が堅牢
- `metadata set` 時に `get_metadata_raw()` で生データを確認し、読めない場合は警告を表示

**リスク:**
- 既存の非 dict メタデータがある場合、`metadata set` で上書きされる
- 警告は表示するが、操作は中断しない（ユーザーの意図を尊重）

### 1.3 タグ同期の best-effort 設計

タグ同期（`_sync_tags`）は本文同期の成功後に実行され、失敗しても本文同期の結果には影響しない。

**理由:**
- タグはメタデータであり、本文の同期が主目的
- タグ API の一時障害で本文同期全体が失敗するのは過剰
- 失敗時は stderr に警告を表示し、debug ログに詳細を記録

## 2. FR-16 ステータス管理

### 2.1 ステータス ID 指定方式

ステータスは eLabFTW のチーム設定で定義される ID で指定する。名前指定は非対応。

**理由:**
- ステータス名はチームごとに異なり、名前→ID の解決にはチーム ID が必要
- eLabFTW API の `GET /teams/{id}/experiments_status` でステータス一覧を取得できるが、チーム ID の取得が前提
- シンプルさを優先し、ID 直接指定とした

### 2.2 一括更新の安全策

`entity-status set` は複数エンティティが対象の場合、確認プロンプトを表示する。`--id` で単体指定も可能。

## 3. FR-17 マルチユーザー対応

### 3.1 whoami の実装範囲

`GET /api/v2/users/me` で現在のユーザー情報を表示する。ロック機構は eLabFTW API にネイティブなロック機能がないため未実装。

### 3.2 同期ログへのユーザー記録

`sync_log.record()` に `user` フィールドを追加（オプション）。後方互換性を維持し、既存のログエントリには影響しない。

## 4. FR-18 テンプレート機能

### 4.1 出力先の決定ロジック

1. `--output` が指定されていればそのパスを使用
2. `--target` で指定されたターゲットの `docs_dir` を使用
3. どちらもなければ最初のターゲットの `docs_dir` を使用
4. 全て `project_root`（設定ファイルの親ディレクトリ）基準で解決

**理由:**
- 既存の `pull` / `sync` と同じパス解決ロジックに統一
- `-c /path/to/.elab-sync.yaml` で別ディレクトリから実行しても正しく動作

### 4.2 ファイル名サニタイズ

テンプレートタイトルから英数字・ハイフン・アンダースコア以外を除去し、スペースをアンダースコアに変換。

**理由:**
- リモートのテンプレートタイトルは任意の文字列を含みうる
- Windows の予約名やパス区切り文字による意図しないファイル配置を防止

### 4.3 HTML → Markdown 変換

`markdownify` の `heading_style="ATX"` を使用。`pull` / `clone` と同じ設定に統一。

## 5. テスト設計思想

### 5.1 全体方針

- eLabFTW API への実通信は行わない（全て `unittest.mock` でモック）
- ファイルシステム操作は `pytest` の `tmp_path` を使用
- 各テストは独立して実行可能（状態を共有しない）

### 5.2 新機能テストのカバレッジ方針

| 機能 | テスト対象 | 重点 |
|---|---|---|
| タグ管理 | add/remove/list + 同期の追記のみ動作 + best-effort | 既存タグを外さないことの検証 |
| メタデータ | get の異常系（null/不正JSON/list型）+ set のマージ動作 | データ保全の境界条件 |
| ステータス | show/set の正常系 + 単体指定 | 一括更新の安全策 |
| テンプレート | 一覧/生成/既存エラー/出力先指定 | パス解決の正確性 |

### 5.3 テストで発見したバグ

- `_get_entity_ids` で `load_item_id`（存在しない）を呼んでいた → `read_item_id` に修正
- テスト追加の価値を実証する事例
