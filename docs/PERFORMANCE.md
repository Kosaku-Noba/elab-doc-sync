# elab-doc-sync 性能仕様書

バージョン: 0.1.0
作成日: 2026-04-05

---

## 1. 概要

本書は elab-doc-sync の性能特性・制約・タイムアウト設定を定義する。

## 2. 通信性能

### 2.1 タイムアウト

| 操作 | タイムアウト | 備考 |
|---|---|---|
| 通常 API リクエスト（GET/PATCH/POST/DELETE） | 30 秒 | `_req()` のデフォルト |
| ファイルアップロード | 60 秒 | 画像サイズに応じて延長 |

### 2.2 リトライ

- 現行実装ではリトライ機構なし。HTTP エラー発生時は即座に `HTTPError` を送出する。
- ネットワーク不安定環境での利用時は、ユーザーが再実行することで対応する（ハッシュ差分検知により、成功済みファイルはスキップされる）。

## 3. 差分検知

### 3.1 ハッシュアルゴリズム

| 項目 | 仕様 |
|---|---|
| アルゴリズム | SHA-256 |
| 保存形式 | 16 進数文字列の先頭 16 文字 |
| 比較対象 | 画像 URL 書き換え前の raw Markdown テキスト |

### 3.2 衝突確率

- 16 文字（64 ビット）のハッシュプレフィックスを使用。
- 研究ドキュメント規模（数百〜数千ファイル）では衝突は実質無視できる。

### 3.3 差分検知の動作

| 状態 | 動作 |
|---|---|
| ハッシュファイルが存在しない | 変更ありと判定（初回同期） |
| ハッシュが一致 | 変更なし → スキップ |
| ハッシュが不一致 | 変更あり → 同期実行 |
| `--force` 指定 | ハッシュ比較をバイパスし無条件で同期 |

## 4. ファイル処理

### 4.1 Markdown 変換

| 項目 | 仕様 |
|---|---|
| 変換ライブラリ | Python-Markdown |
| 有効な拡張 | `tables`, `fenced_code`, `codehilite`, `toc`, `nl2br` |
| 逆変換（pull 時） | markdownify（`heading_style="ATX"`） |

### 4.2 画像処理

| 項目 | 仕様 |
|---|---|
| 検出パターン | `!\[([^\]]*)\]\(([^)]+)\)` |
| スキップ条件 | URL が `http://` または `https://` で始まる場合 |
| パス解決順序 | 1. `docs_dir` 相対 → 2. `project_root` 相対 |
| アップロード先 | eLabFTW の `/api/v2/{entity_type}/{entity_id}/uploads` |
| 失敗時の動作 | 警告出力、元の参照を維持（同期は中断しない） |

### 4.3 ファイル結合（merge モード）

| 項目 | 仕様 |
|---|---|
| ファイル順序 | ファイル名の辞書順（`sorted()`） |
| セパレータ | `\n\n---\n\n`（水平線） |

## 5. API 通信仕様

### 5.1 エンドポイント

| 操作 | メソッド | パス |
|---|---|---|
| アイテム一覧 | GET | `/api/v2/items` |
| アイテム取得 | GET | `/api/v2/items/{id}` |
| アイテム作成 | POST | `/api/v2/items` |
| アイテム更新 | PATCH | `/api/v2/items/{id}` |
| アイテム削除 | DELETE | `/api/v2/items/{id}` |
| 実験一覧 | GET | `/api/v2/experiments` |
| 実験取得 | GET | `/api/v2/experiments/{id}` |
| 実験作成 | POST | `/api/v2/experiments` |
| 実験更新 | PATCH | `/api/v2/experiments/{id}` |
| 実験削除 | DELETE | `/api/v2/experiments/{id}` |
| ファイルアップロード | POST | `/api/v2/{entity_type}/{entity_id}/uploads` |
| タグ追加 | POST | `/api/v2/{entity_type}/{entity_id}/tags` |
| タグ削除 | DELETE | `/api/v2/{entity_type}/{entity_id}/tags/{tag_id}` |
| メタデータ更新 | PATCH | `/api/v2/{entity_type}/{entity_id}` |

### 5.2 認証

| 項目 | 仕様 |
|---|---|
| 方式 | `Authorization` ヘッダーに API キーを直接設定 |
| Content-Type | `application/json`（ファイルアップロード時は `multipart/form-data`） |

### 5.3 エンティティ ID の取得

- 作成レスポンスの JSON `id` フィールドを優先。
- 取得できない場合は `Location` ヘッダーの末尾パスセグメントから抽出。

## 6. 同期フロー

### 6.1 Push（merge モード）

```
1. docs_dir から pattern に一致するファイルを収集（辞書順）
2. 全ファイルを結合
3. SHA-256 ハッシュを計算し、保存済みハッシュと比較
4. 変更なし → スキップ（--force 時は続行）
5. ID ファイルからエンティティ ID を読み込み
6. ID なし or エンティティ不在 → 新規作成
7. ローカル画像をアップロード・URL 書き換え
8. Markdown → HTML 変換
9. eLabFTW に PATCH で更新
10. ハッシュを保存
```

### 6.2 Push（each モード）

```
1. docs_dir から pattern に一致するファイルを収集
2. mapping.json からファイル名 → ID のマッピングを読み込み
3. 各ファイルについて:
   a. SHA-256 ハッシュで差分検知
   b. 変更なし → スキップ
   c. マッピングに ID なし or エンティティ不在 → 新規作成
   d. 画像アップロード・URL 書き換え
   e. Markdown → HTML 変換
   f. eLabFTW に PATCH で更新
   g. ハッシュ・マッピングを保存
```

### 6.3 Pull

```
1. 対象エンティティを特定（--id / mapping / 全件取得）
2. eLabFTW から HTML body を取得
3. HTML → Markdown に変換（markdownify）
4. ローカルにファイル保存
5. ID マッピング・ハッシュを保存（次回 push 時の不要更新を防止）
```

## 7. 制約・制限事項

| 項目 | 制限 |
|---|---|
| 同時接続 | シングルスレッド・逐次処理 |
| ファイルサイズ上限 | eLabFTW サーバー側の設定に依存 |
| 対応画像形式 | eLabFTW がアップロードを受け付ける全形式 |
| Markdown 方言 | Python-Markdown 準拠（GFM とは一部差異あり） |
| 双方向同期 | 非対応（push と pull は独立操作、競合解決機能なし） |
| リトライ | 未実装（再実行で対応） |
| レート制限 | 未実装（eLabFTW 側の制限に依存） |

## 8. 将来の改善候補

| 項目 | 概要 |
