# マルチエージェント開発フロー仕様書（v1.1 Draft）

## 1. 文書情報

**文書名**
マルチエージェント開発フロー仕様書

**版数**
v1.1（指揮AI除外版）

**目的**
AIを用いた開発プロセスにおいて、

* 要求整理
* 実装
* コード品質確認
* ドキュメント品質確認

を分離し、**再現性・可読性・拡張性の高い開発フロー**を定義する。

---

## 2. 背景

従来のAI開発では、

* 要求が曖昧なまま実装に入る
* 実装の意図が記録されない
* コードレビューとドキュメントレビューが混在する
* AIが自分の出力を正しい前提で進める

といった問題がある。

本仕様では、これを解決するために、

> **「提案 → 実装 → コード検証 → 説明検証」**

の4段階に分離する。

---

## 3. システム概要

### 3.1 構成

```text
User
 ↓
Feature Proposal Agent
 ↓
Kiro（実装）
 ↓
git commit
 ↓
Codex-1（コードレビュー）
 ↓
Codex-2（ドキュメントレビュー）
 ↓
AI_discussions.md（記録）
 ↓
Kiro（修正）
```

---

## 4. コンポーネント仕様

### 4.1 Feature Proposal Agent（機能提案AI）

#### 役割

ユーザーの要求を**実装可能な仕様に整理する**

#### 主な責務

* 要求の分解
* ユースケース抽出
* 機能候補の提示
* MVPと後回しの切り分け
* 不明点の洗い出し

#### 出力

```markdown
## [FeatureProposal]

### 要求の整理
- ...

### 提案機能
- ...

### MVP範囲
- ...

### 不明点
- ...
```

#### 特徴

* コードには触れない
* 上流工程専用
* 誤っても後工程で修正可能（低リスク）

---

### 4.2 Kiro（実装担当）

#### 役割

仕様に基づきコードを実装する

#### 主な責務

* コード変更
* 変更理由の記録
* レビュー指摘への対応

#### 出力

```markdown
## [Kiro]

### 変更内容
- ...

### 理由
- ...

### 影響範囲
- ...
```

---

### 4.3 Codex-1（コードレビュー）

#### 役割

コードの正しさ・安全性を確認する

#### 主な責務

* バグ検出
* 例外処理確認
* 境界条件確認
* テスト不足検出
* 設計の破綻検出

#### 出力

```markdown
## [CodeReview]

### 指摘事項
- [P1] ...
- [P2] ...
- [P3] ...

### 結論
- 修正推奨 / 問題なし
```

---

### 4.4 Codex-2（ドキュメントレビュー）

#### 役割

人間向け説明の品質を確認する

#### 主な責務

* README更新漏れ検出
* 仕様説明不足検出
* コメント不足検出
* 初見理解性チェック

#### 出力

```markdown
## [DocReview]

### 指摘事項
- [P1] ...
- [P2] ...
- [P3] ...

### 結論
- 文書更新推奨 / 問題なし
```

---

### 4.5 AI_discussions.md（共有記録）

#### 役割

全エージェントのやり取りを記録する

#### 内容

* 仕様提案
* 実装内容
* レビュー結果
* 修正履歴

#### 特徴

* 人間が後追い可能
* 監査ログとして機能

---

## 5. 機能要件

### 5.1 要求整理

* Feature Proposal Agent が仕様を生成できること

### 5.2 実装

* Kiro が仕様に基づきコードを変更できること

### 5.3 コードレビュー

* Codex-1 が差分レビューできること

### 5.4 ドキュメントレビュー

* Codex-2 が説明不足を検出できること

### 5.5 記録

* 全工程が `AI_discussions.md` に記録されること

---

## 6. 非機能要件

### 可読性

* 初見でも履歴が追える

### 拡張性

* エージェント追加が容易

### 安全性

* レビュー失敗でも開発は止めない（fail-open）

### 再現性

* 同じ入力で同様の結果が得られる

## 7. ユースケース

---

### UC-01: 新機能追加

#### シナリオ

ユーザーが「ログイン履歴表示機能が欲しい」と依頼

#### フロー

1. Feature Proposal Agent

   * ログイン履歴機能を分解
   * MVP定義

2. Kiro

   * DB追加
   * API追加
   * UI実装

3. Codex-1

   * セキュリティチェック（履歴漏洩）
   * 例外処理確認

4. Codex-2

   * README更新漏れ検出
   * API仕様未記載指摘

5. Kiro

   * 修正対応

---

### UC-02: バグ修正

#### シナリオ

「ログイン時に500エラーが出る」

#### フロー

1. Feature Proposal Agent

   * 原因仮説整理
   * 修正方針提案

2. Kiro

   * 修正実装

3. Codex-1

   * 他のエラーケース漏れ指摘

4. Codex-2

   * エラー仕様未記載指摘

---

### UC-03: ドキュメント改善

#### シナリオ

READMEが分かりにくい

#### フロー

1. Feature Proposal Agent

   * 改善項目整理

2. Kiro

   * README修正

3. Codex-2

   * 初見理解性チェック

（Codex-1はスキップ可）

---

### UC-04: 軽微修正

#### シナリオ

文言修正

#### フロー

* Feature Proposal: スキップ可
* Codex-1: スキップ可
* Codex-2: 任意実行

---

## 8. 制約事項

* Feature Proposal はコードを書かない
* Codex-2 はコードレビューを主目的にしない
* AI_discussions.md以外を勝手にamendしない
* エージェントの責務を重複させない

## 10. まとめ

本仕様は、

> **「AIを役割分担させて開発プロセスを構造化する」**

ことを目的とする。

特に重要なのは以下である。

* 提案と実装を分離する
* コードとドキュメントのレビューを分離する
* すべての判断を記録する

この構成により、

* AI開発のブラックボックス化を防ぎ
* 品質と説明性を両立し
* 将来の拡張にも耐える

開発基盤を実現する。
